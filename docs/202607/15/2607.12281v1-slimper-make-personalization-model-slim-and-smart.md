## 摘要
Transformer风格的架构越来越多地被用于工业推荐系统，但它们继承了一个与任务不匹配的设计前提：生成模型依赖于逐个token的自回归预测，这证明了保持随序列长度扩展的大型中间张量是合理的。相比之下，推荐系统为每个<用户, 物品>对生成一组相关性分数，而没有token级别的监督。利用这一观察，我们提出了SlimPer，它将个性化排名重新表述为对紧凑、统一的<用户, 物品>知识库的迭代优化。在每一层，模型有选择地查询原始的多模态用户侧token，计算显式的相关性匹配分数，并优化知识库，所有操作都以固定的中间表示大小和每层O(N)的代价完成。因此，模型深度与用户历史长度解耦，使得在计算或内存不成比例增长的情况下实现更深层次的相关性理解；仅请求优化通过在所有候选物品之间共享用户侧token的单一副本进一步减少内存。SlimPer在单一骨干网络中统一了稀疏、稠密和序列特征，并通过其注意力机制提供固有的可解释性。部署在Instagram Reels和Feed上后，SlimPer在提升用户参与度的同时简化了整体系统，并能够有效建模超过1万个细粒度用户历史事件。

## Abstract
Transformer-style architectures are increasingly adopted for industrial recommendation systems, yet they inherit a design premise misaligned with the task: generative models rely on per-token autoregressive prediction, which justifies maintaining large intermediate tensors that scale with sequence length. In contrast, recommendation systems produce a single set of relevance scores for each <user, item> pair without token-level supervision. Leveraging this observation, we propose SlimPer, which reformulates personalized ranking as iterative refinement of a compact, unified <user, item> knowledge base. At each layer, the model selectively queries raw multi-modal user-side tokens, computes explicit relevance matching scores, and refines the knowledge base, all in O(N) per-layer cost with a fixed-size intermediate representation. As a result, model depth is decoupled from user history length, enabling deeper relevance understanding without proportional growth in compute or memory; request-only optimization further trims memory by sharing a single copy of user-side tokens across all candidate items. SlimPer unifies sparse, dense, and sequence features within a single backbone and provides inherent interpretability through its attention mechanism. Deployed on Instagram Reels and Feed, SlimPer yields measurable improvements in user engagement while streamlining the overall system and enabling effective modeling of 10k+ fine-grained user history events.

### 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：将个性化排序形式化为对固定大小的<用户，物品>知识库（Knowledge Base, KB）的迭代精炼。每一层都执行三步操作：（1）**选择**（Select）：通过全模态感知的QKV注意力从原始用户侧token中检索相关证据；（2）**匹配**（Match）：对检索到的证据与多样板（templates）计算显式点积相关性分数；（3）**精炼**（Refine）：利用匹配结果更新知识库。整个过程中，知识库尺寸固定（如K=64），每层计算复杂度为O(N)，模型深度L与序列长度N完全解耦。
- **关键技术细节**：
  - **知识库初始化**：从非用户侧（物品/交叉侧）token经线性投影得到初始KB₀ ∈ ℝ^(K×d)。
  - **Tokenization模块**：
    - 稀疏特征：通过Embedding Pooling（均值或求和池化）得到固定大小向量。
    - 序列事件：每个事件编码为1×d token，融合内容、动作类型、时间信息及**上下文编码**（通过滑动窗口聚合邻居事件信息，增强连续性）。
    - 稠密特征：拼接后经MLP投影为固定向量。
  - **SlimPer层（每层）**：
    1. **Select**：从当前知识库X_k生成查询Q（线性投影，Q∈ℝ^(q×d)）。所有用户侧token（稀疏token S和序列token E）作为Keys和Values。对于稀疏特征，使用轻量MLP注意力核；对于序列特征，使用标准缩放点积注意力（Softmax(γQK^T)）。输出检索到的表示R_s和R_e（均为q×d）。
    2. **Match**：从知识库X_k线性投影得到多样板T∈ℝ^(t×d)。分别计算R_s与T、R_e与T的点积，得到匹配分数λ_s和λ_e（均为q×t）。
    3. **Refine**：将匹配分数经RMSNorm后，与X_k的线性投影、稠密特征D拼接，再通过MLPμ生成更新量，与X_k相加得到下一层X_{k+1}。每层还可输出任务嵌入用于多任务预测。
  - **Request-Only Optimization (ROO)**：所有用户侧token（包括稀疏、序列、稠密）在一次请求中只计算一次，并在同一请求的所有候选物品间共享，大幅降低内存和重复计算。
  - **模型可解释性**：注意力权重直接关联用户历史中的特定事件，可生成热力图展示由粗到细的聚焦过程。

### 7. 优点

- **根本性解决了Transformer在推荐中的效率-质量矛盾**：通过固定大小知识库和迭代精炼，实现模型深度与历史长度完全解耦，使得在长序列下能维持更低内存和计算。
- **统一的多模态融合**：单一骨干网络同时处理稀疏、稠密和序列特征，消除了传统方法中分组件分别处理再浅层拼接的冗余和信息损失。
- **请求级优化（ROO）的全模型延伸**：扩展到所有用户侧特征，而不仅仅是历史序列，最大化计算复用，降低内存占用。
- **内在可解释性**：注意力机制自然提供归因，有助于调试、审计和模型分析，这在工业系统中非常实用。
- **信息论论证**：论证知识库瓶颈并不损失任务相关信息，反而引导容量聚焦于用户-物品相关性，优于Transformer中大量O(N²)的用户侧-用户侧计算。
- **实际部署验证**：在Meta两大核心产品上全流量上线，获得显著业务指标提升，证明了方法的实际有效性。
